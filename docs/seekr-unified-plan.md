## 3. 개발자가 해야 할 일

> 상세 일정은 `seekr-dev-schedule.md` 참조. 여기서는 개발 범위와 설계 원칙만 기술.

### 3.1 개발 범위 요약

| Phase | 기간 | 개발 목표 | 핵심 산출물 |
|---|---|---|---|
| **Phase 0** | W1-3 | MCP 서버 3개 + 코어 엔진 | DART/카카오톡/네이버웍스 MCP, FastAPI 서버, Docker |
| **Phase 1** | W4-6 | SDK + 통합 서버 + 확장 | Python SDK(PyPI), OAuth2, GitHub/Gmail, 토스/쿠팡 |
| **Phase 2** | W7-8 | 프로덕션 강화 + 안정화 | 캐싱, 감사로그, RBAC, 테스트 80%+, CI/CD |
| **Phase 3** | W9-10 | 문서 + 런칭 지원 | 문서 사이트, 예제 3개, 플레이그라운드 UI, v1.0.0 |

### 3.2 설계 원칙 (20개 프레임워크 공통 문제 대응)

Seekr는 에이전트 **프레임워크가 아닌 도구 인프라**. 이 포지셔닝이 7가지 공통 문제 중 4가지를 구조적으로 회피한다.

| 공통 문제 (빈도) | 전략 | 핵심 설계 |
|---|---|---|
| 무한 루프 (14/20) | 구조적 회피 + 방어 기여 | `ExecutionGuard` — 연속 호출 감지/차단 |
| 토큰 비용 (16/20) | 3중 방어 | Redis 캐싱(TTL) + 결과 압축 + 비용 대시보드 |
| 보안 취약점 (8/20) | 공격면 제거 | 코드 실행 없음, `SafeHttpClient`(SSRF 차단), Pydantic 검증, AES 암호화 |
| 메모리 누수 (10/20) | 무상태 서버 | 요청 스코프 GC, 모든 상태 외부 DB |
| LLM 호환성 (17/20) | 문제 자체 회피 | LLM 비호출 원칙, `ToolRegistry`가 MCP/OpenAI/LangChain 스키마 내보내기 |
| 프로덕션 부족 (15/20) | Day 1 설계 | 감사 로그, RBAC, OpenTelemetry, `/health` |
| Breaking Changes (12/20) | SemVer + API 버전 | `/v1/`, Deprecation 정책, 하위 호환성 보장 |

### 3.3 기술 스택

| 영역 | 선택 | 근거 |
|---|---|---|
| 백엔드 | FastAPI (Python) | Pydantic 네이티브, 비동기, 한국 채택률 |
| SDK | Python → TypeScript (Phase 3+) | Composio 패턴 |
| DB | PostgreSQL | 엔터프라이즈 표준, pgvector 확장 |
| 캐시 | Redis | TTL 캐싱, 속도 제한, 세션 |
| 관측성 | OpenTelemetry | 벤더 중립 |
| 컨테이너 | Docker Compose | 원클릭 셀프호스팅 |
| 프로토콜 | MCP | 프레임워크 비종속 핵심 |

## 5. 비즈니스 모델

| 티어 | 가격 | 대상 |
|---|---|---|
| **Community** | 무료 (Apache 2.0) | 개인 개발자, 스타트업 |
| **Pro** | ₩49,000/월 | 소규모 팀 (고급 통합, 우선 지원) |

수익 모델: 오픈소스(채택) + 관리형 클라우드(편의) + 엔터프라이즈(수익)
